# SBATCH脚本内存监控使用说明

## 概述

为了更好地了解GRASP计算过程中的内存使用情况，我们提供了几种内存监控方法，可以持续记录和分析内存使用量。

## 方法一：基于 `ps` 和 `free` 的内存监控（推荐）

### 特点
- ✅ 兼容性好，适用于所有Linux系统
- ✅ 可以分别监控系统内存和特定进程内存
- ✅ 实时监控，间隔可调
- ✅ 自动生成详细报告

### 使用方法
```bash
# 在您的脚本中添加以下代码
MEMORY_LOG="${SLURM_JOB_ID}_memory_usage.csv"
start_memory_monitor "$MEMORY_LOG" 20  # 每20秒记录一次

# 设置退出陷阱，确保监控正常结束
trap 'stop_memory_monitor; generate_memory_report "$MEMORY_LOG"' EXIT
```

### 输出文件
1. **CSV数据文件**: `{JOB_ID}_memory_usage.csv`
   - 包含时间序列的详细内存数据
   - 格式：时间,系统总内存,系统已用内存,系统可用内存,内存使用率,GRASP进程内存,GRASP进程CPU,Python进程内存

2. **报告文件**: `{JOB_ID}_memory_usage_report.txt`
   - 包含统计摘要和分析建议
   - 显示最大值、平均值等关键指标

## 方法二：SLURM内置内存监控

### 特点
- ✅ 与SLURM深度集成
- ✅ 提供作业级别的资源统计
- ✅ 可以与 `seff` 和 `sacct` 命令配合使用
- ⚠️ 需要SLURM环境支持

### 使用方法
```bash
#SBATCH --mem=64G                    # 设置内存限制
#SBATCH --time=24:00:00             # 设置时间限制

# 启动SLURM内存监控
monitor_slurm_memory 60  # 每60秒监控一次
```

### 查看结果
```bash
# 查看作业统计
sacct -j $SLURM_JOB_ID --format=JobID,JobName,MaxRSS,AveRSS,MaxVMSize,ElapsedTime,State --units=M

# 查看作业效率
seff $SLURM_JOB_ID
```

## 方法三：集成到现有GRASP脚本

我们提供了一个完整的集成版本 `grasp_with_memory_monitor.sh`，它包含：

### 功能特点

- 🐍 监控Python机器学习训练过程的内存使用
- 📊 每个程序执行前后记录内存状态
- 📈 生成详细的内存使用报告
- 🚨 与现有错误检查机制完美集成

### 监控的程序类型

## 内存监控最佳实践

### 1. 监控间隔设置
```bash
# 对于快速程序（<5分钟）
start_memory_monitor "$MEMORY_LOG" 5   # 5秒间隔

# 对于中等程序（5-30分钟）
start_memory_monitor "$MEMORY_LOG" 20  # 20秒间隔

# 对于长时间程序（>30分钟）
start_memory_monitor "$MEMORY_LOG" 60  # 60秒间隔
```

### 2. 内存限制设置
```bash
# 根据预期使用量设置合理的内存限制
#SBATCH --mem=32G   # 对于小型计算
#SBATCH --mem=64G   # 对于中型计算
#SBATCH --mem=128G  # 对于大型计算
```

### 3. 数据分析命令
```bash
# 查看内存使用趋势
cut -d',' -f1,3 ${JOB_ID}_memory_usage.csv | tail -20

# 查看GRASP程序内存峰值
cut -d',' -f1,6 ${JOB_ID}_memory_usage.csv | sort -t',' -k2 -n | tail -5

# 查看Python程序内存使用
cut -d',' -f1,8 ${JOB_ID}_memory_usage.csv | grep -v ",0"
```

## 故障排除

### 常见问题
1. **监控进程未启动**
   - 检查 `bc` 命令是否可用：`which bc`
   - 检查权限：确保可以写入 `/tmp` 目录

2. **数据文件为空**
   - 检查进程名匹配：确保监控的进程名正确
   - 检查监控间隔：可能程序运行时间太短

3. **内存数据异常**
   - 检查系统负载：`top` 或 `htop`
   - 检查磁盘空间：`df -h`

### 调试方法
```bash
# 检查监控进程是否运行
ps aux | grep memory_monitor

# 手动测试内存获取
free -g | grep "Mem:"

# 检查进程查找
pgrep -f "rmcdhf|rci|python"
```

## 性能影响

内存监控本身的资源消耗：
- **CPU使用**: < 0.1%
- **内存使用**: < 10MB
- **磁盘IO**: 每次记录约100字节

对计算性能的影响可以忽略不计。

## 总结

推荐使用**方法一**（基于ps和free的监控）作为标准方案，因为它：
- 兼容性最好
- 功能最全面
- 输出最详细
- 易于定制和扩展

对于有SLURM环境的用户，可以同时使用**方法二**获得官方的资源统计数据作为补充。
